# Prose Diff 与合并冲突

InkStream 的版本对比和冲突解决可以按句子来看，而不只是按行。你能看清哪一句话被改了；合并出现冲突时逐处采纳本方或对方的写法，不用手动删冲突标记。

## 在哪里看 diff

1. 打开 Git Graph 视图：按 `Ctrl+Shift+G`，或点左下角 git 面板里的入口。当前工作区不是 git 仓库时会提示「当前工作区不是 git 仓库」，不会打开。
2. 在左栏点选一个提交，中栏会列出该提交改动的文件。
3. 点中栏里的某个文件，右栏就会显示这次改动的对比内容。

## 行 diff 与句 diff 切换

右栏顶部有一个「行 / 句」切换按钮，对应两种看法：

- 行：传统的逐行对比，新增行绿底、删除行红底，左侧带新旧行号，等宽字体显示。适合看代码或精确的行级改动。
- 句：句级语义对比（prose diff）。把改动按句子拆开，只把真正变动的句子高亮出来。

切换只影响右栏的呈现方式，不改动文件本身。

## 句级 prose diff 怎么读

句 diff 面向长文写作，回答的是「哪句话改了」而不是「哪行变了」：

- 绿色高亮：新增的句子。
- 红色高亮加删除线：删掉的句子。
- 无高亮：未改动的句子，原样显示。
- 一句话被改写时，会成对出现「删除的旧句 + 新增的新句」。

要点：

- 中英文混排都能正确断句（中文按「。！？」、英文按「.!?」）。
- 只是空格、换行或缩进的变化不算改动，不会被标红标绿。也就是说，重排版式但内容没变时，句子仍显示为未改。
- 段落结构会保留，不同段落之间留出间距分隔。
- 二进制文件显示「二进制文件，不显示 diff」；没有文本变更时显示「无文本变更」。

## 合并冲突会怎样

执行合并、cherry-pick 或撤销提交（revert）时，如果 git 无法自动合并，会弹出一条警告提示，告诉你产生了冲突。点击这条提示即可打开「prose 三向解决器」，逐处采纳本方或对方，也可以在解决器里中止本次操作。

提示：可自动合并的部分 git 已经帮你并好，解决器里只会列出真正冲突的地方。

## 用三向解决器解决冲突

解决器以整页视图打开，左栏列出所有待解决的冲突文件，右栏显示选中文件的内容。打开时不会抢走编辑器的输入焦点。

右栏把文件内容拆成两类：

- 干净段：git 已经合好的部分，以灰色原样显示，不需要你处理。
- 冲突块：每一处真冲突显示为一张带边框的卡片，标题为「冲突（本方 ↔ 对方）」，卡片内用句级对比展示本方与对方的差异。

对每个冲突块，卡片右上角有三个按钮，逐块选择：

- 采纳本方：保留你这一侧的内容（默认选中）。
- 采纳对方：采用合并进来的另一侧内容。
- 两者都要：把本方和对方的内容都保留下来。

顶部状态条会显示「N 个文件待解决」，右栏底部会显示当前文件有「N 处冲突」。

## 保存、标记解决与提交

- 为当前文件的每个冲突块选好之后，点右栏底部的「保存并标记解决」。系统会按你的选择组装出最终内容写回文件并加入暂存（标记为已解决），该文件随即从左栏冲突列表中消失。
- 逐个文件解决，直到列表清空。全部解决后会提示「全部冲突已解决，请在左下角 git 面板提交合并结果」，此时回到 git 面板提交即可完成本次合并。
- 注意：解决器只负责把冲突标记处理掉，它不会自动提交。最终需要你自己提交。提交时若仍检测到未解决的冲突，会先二次确认，避免把 `<<<<<<<` 等冲突标记误提交进历史。

## 中止合并

如果不想继续这次合并（或 cherry-pick / revert），点解决器顶部的「中止合并」即可放弃本次操作。中止成功后会自动回到编辑器；若中止失败则停留在解决器并提示错误。顶部右侧的关闭按钮只是关掉解决器回到编辑器，并不会中止合并。
